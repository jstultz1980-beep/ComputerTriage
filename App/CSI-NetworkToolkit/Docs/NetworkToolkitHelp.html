Purpose And Mindset

The Network Toolkit is intended to be a portable, technician-friendly troubleshooting console. The goal is not to replace good judgment. The goal is to make the first pass fast, consistent, and easy for a junior technician to run without memorizing dozens of command-line switches.

The toolkit is built around live troubleshooting. It should expose what is broken right now, collect evidence that helps the next technician, and avoid unnecessary historical clutter. The current design uses a limited latest-state model: the toolkit can keep the newest useful discovery data per computer, while older temporary outputs can be removed with the sanitize option.

Best first move: run Quick Diagnosis, review the report, then use the recommended tab or tool for deeper investigation. Do not start repair tools before you understand what the report is pointing at.
Portable toolkit rule: this toolkit may be running from a flash drive. Avoid writing unnecessary long-term data to the toolkit. Use Reports, Computer Info, and TempToolOutputs for active troubleshooting evidence, then sanitize when done.

Launching The Toolkit

Normal GUI launch

Use the main launcher in the portable root. The GUI is the primary experience for technicians. It hides console noise where possible, organizes tools by purpose, and opens external utilities with the correct working paths.

LauncherUse
App\NetworkToolkit.ps1Main PowerShell launcher. Opens the GUI by default. Use -CLI to open the console toolkit.
NetworkToolkit.vbsPrimary portable-root launcher. It starts the toolkit elevated while keeping the PowerShell host window hidden.
App\CSI-NetworkToolkit\CSI-NetworkToolkit.ps1Console toolkit backend. Preserved for command-line use and for GUI button actions that still run console-driven tools.

Elevation

Some tools work fine as a standard user. Others need local administrator rights. The header shows whether the toolkit is running elevated. If the task involves system repair, Windows Update repair, service manipulation, driver inspection, packet capture, security scanners, stale printer cleanup, or some Sysinternals tools, elevation may be required.

If a tool fails with access denied, restart the toolkit elevated before assuming the computer itself is broken.

Quick Diagnosis

Quick Diagnosis is the main walk-up workflow. The idea is simple: plug in the toolkit, launch it, run one task, and get a professional status report that tells the technician what is healthy, what needs review, and what to do next.

Main buttons

The report is not automatically opened unless Settings says to do so.

Quick target checks

The Quick Diagnosis page also includes quick target fields for common network checks. These are for immediate troubleshooting while you are standing at the machine.

How to read the Quick Diagnosis report

The report is intentionally split into summary, system snapshot, disks, network adapters, problems found, all checks, and driver inventory. The important design choice is that passing checks should stay compact. If a check passes, the report should show the task name and an Error free style indicator. Supporting evidence is only expanded for warnings or critical issues.

StatusMeaningAction
OKThe check did not find an obvious issue.No immediate follow-up unless the user's symptom points directly at that area.
InfoThe toolkit collected useful context or could not confirm a signal. This is not necessarily a problem.Review only if related to the symptom. Virtual machines often produce Info findings for hardware checks.
WarningThe toolkit found something that may be broken, misconfigured, blocked by permission, or worth checking.Read the supporting evidence and recommended next step. Use the suggested tab/tool to investigate.
CriticalThe toolkit found a high-risk condition.Stop and investigate before making changes. Back up data first if storage failure is suspected.
Virtual machine note: hardware tools such as SMART, CrystalDiskInfo, GPU inspection, and some sensor checks may be limited in VMs. If the toolkit says storage visibility is limited, check the hypervisor host or storage platform as well as the guest.

Reports should help answer questions like: what computer did I check, what did Quick Diagnosis find, what profile was captured, what network discovery output exists, and where are the repair records?

Computer Info

The Computer Info tab is the technician-facing view for saved computer profiles. A computer profile is the evolved version of the original computer fingerprint. It captures identity, OS, hardware, network configuration, local administrators, listening ports, security products, Defender, BitLocker, servicing health, pending reboot state, and toolkit latest-state data.

ButtonPurpose
ViewOpens the selected computer profile as a readable HTML report.
Take ProfileCaptures the current computer profile and saves it for later selection.
DeleteDeletes the selected saved computer profile after confirmation.
RefreshReloads the saved profile list.

The Computer Profile report should become more useful over time because it can include latest Quick Diagnosis data and latest GUI tool outputs from the per-computer state file.

Software Keys Tab

Software Keys is a read-only local licensing review tool. It checks Windows firmware and registry licensing, legacy Microsoft Office registration, and explicit application registry values named as product, serial, license, registration, or activation entries. It does not retrieve passwords, browser data, Wi-Fi keys, API tokens, or password/token-style registry values.

ControlPurpose
Scan This ComputerCollects recoverable local licensing and registration entries and refreshes both grids.
Copy Selected KeyCopies the selected value only after a technician selects a row.

Many modern applications use cloud accounts, subscriptions, encrypted stores, or volume activation. A missing key does not mean the software is unlicensed; it means no recoverable local key was found.

Live Log And Troubleshooting Toolkit Problems

The Live Log view is opened from Settings and shows high-level GUI activity and errors. It is not meant to capture every line of every tool. Each tool also has a small usage log under CSI-NetworkToolkit\Logs\ToolUsage.

ButtonPurpose
Clear LogClears the visible live log box.
Copy LogCopies the live log text to the clipboard.

When a button does nothing

  1. Check the status bar for a short failure message.
  2. Open Settings, then View Live Log, and look for a recent error.
  3. Open Settings, then Open Logs, and inspect the matching ToolUsage log.
  4. Confirm the portable app still exists in CSI-NetworkToolkit\Custom or ExternalTools.
  5. If the tool requires elevation, restart the toolkit elevated.

Recommended Field Workflows

Walk-up workstation triage

  1. Launch the toolkit elevated if possible.
  2. Verify computer name, domain, private IP, and elevation in the header.
  3. Run Quick Diagnosis.
  4. Open Latest Report.
  5. Review Problems Found first. Ignore passing checks unless the symptom points to that area.
  6. Use the recommended tab/tool for each warning.
  7. If repairs are needed, run targeted repair tools only after reviewing evidence.
  8. Save or export anything useful, then sanitize the toolkit when the engagement is complete.

Network issue workflow

  1. Check private IP, adapter, gateway, and DNS in the header/report.
  2. Use Quick target Ping/TCPing against gateway, DNS, domain controller, and internet target.
  3. Run Connectivity Triage and Adapter Route Health.
  4. Use DNS Path Test if names fail but IPs work.
  5. Use Live Route Trace or WinMTR for intermittent latency/loss.
  6. Use Wireshark when DHCP/DNS/TCP behavior needs packet proof.

BSOD or crash workflow

  1. Run Quick Diagnosis and check crash, driver, hardware, and disk findings.
  2. Run Minidump Collector.
  3. Open BlueScreenView against the collected dumps.
  4. Open Reliability Monitor for timeline context.
  5. Compare crash time with driver events and Windows Update history.
  6. Update or roll back drivers only when evidence points to them.

Windows Update broken workflow

  1. Run Quick Diagnosis and Windows Update Health.
  2. Open Windows Update tab and Refresh Updates.
  3. Review pending updates and history result codes.
  4. Use Download Selected if you want to stage an update without installing.
  5. Use Repair Windows Update only when scan/install/update components are broken.
  6. Reboot if repair or update status indicates it, then refresh again.
