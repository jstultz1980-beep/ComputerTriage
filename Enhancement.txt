would it be possible to make a tool that uses choco to add portable tools to the toolkit? They could just go on a "Custom" tab for now.

Yes. Chocolatey can install many packages with --install-arguments, but for a portable toolkit the better pattern is:

Use Chocolatey only as the package resolver/downloader/installer, then copy or extract the resulting portable app into your toolkit’s Custom folder.

A simple structure:

Toolkit\
  ToolkitLauncher.exe / .ps1
  Apps\
    PortableApps\
    Sysinternals\
    NirSoft\
  Custom\
    Everything\
    WinMerge\
    Notepad++\
    SystemInformer\
  manifests\
    custom-tools.json
Chocolatey-backed add flow:

1. User enters package name
2. Tool runs: choco info <package>
3. User confirms
4. Tool installs package to temp/cache path
5. Tool detects EXE/folder output
6. Tool copies portable files into .\Custom\<ToolName>
7. Tool adds metadata to Custom tab manifest
8. Custom tab refreshes
Example PowerShell prototype:

param(
    [Parameter(Mandatory)]
    [string]$PackageName,

    [string]$ToolkitRoot = "$PSScriptRoot",
    [string]$CustomRoot = "$PSScriptRoot\Custom"
)

$ErrorActionPreference = "Stop"

if (-not (Get-Command choco.exe -ErrorAction SilentlyContinue)) {
    throw "Chocolatey is not installed or not in PATH."
}

New-Item -ItemType Directory -Force -Path $CustomRoot | Out-Null

$workRoot = Join-Path $env:TEMP "ToolkitChocoInstall"
$pkgRoot  = Join-Path $workRoot $PackageName

Remove-Item $pkgRoot -Recurse -Force -ErrorAction SilentlyContinue
New-Item -ItemType Directory -Force -Path $pkgRoot | Out-Null

Write-Host "Installing $PackageName to temporary Chocolatey cache..."

choco install $PackageName `
    --yes `
    --no-progress `
    --limit-output `
    --install-directory $pkgRoot

$dest = Join-Path $CustomRoot $PackageName

Remove-Item $dest -Recurse -Force -ErrorAction SilentlyContinue
New-Item -ItemType Directory -Force -Path $dest | Out-Null

# Look for likely installed files
$possibleSources = @(
    $pkgRoot,
    "C:\ProgramData\chocolatey\lib\$PackageName",
    "C:\ProgramData\chocolatey\bin"
) | Where-Object { Test-Path $_ }

foreach ($src in $possibleSources) {
    Get-ChildItem $src -Recurse -File -ErrorAction SilentlyContinue |
        Where-Object {
            $_.Extension -in ".exe", ".bat", ".cmd", ".ps1", ".dll", ".json", ".config", ".ini"
        } |
        ForEach-Object {
            $relative = $_.FullName.Substring($src.Length).TrimStart('\')
            $target = Join-Path $dest $relative
            New-Item -ItemType Directory -Force -Path (Split-Path $target) | Out-Null
            Copy-Item $_.FullName $target -Force
        }
}

$exe = Get-ChildItem $dest -Recurse -File -Include *.exe |
    Sort-Object Length -Descending |
    Select-Object -First 1

$manifestPath = Join-Path $ToolkitRoot "custom-tools.json"

if (Test-Path $manifestPath) {
    $manifest = Get-Content $manifestPath -Raw | ConvertFrom-Json
} else {
    $manifest = [pscustomobject]@{
        tools = @()
    }
}

$entry = [pscustomobject]@{
    name        = $PackageName
    source      = "Chocolatey"
    package     = $PackageName
    path        = if ($exe) { $exe.FullName.Replace($ToolkitRoot, ".") } else { ".\Custom\$PackageName" }
    installed   = (Get-Date).ToString("s")
    category    = "Custom"
}

$manifest.tools = @($manifest.tools | Where-Object { $_.package -ne $PackageName }) + $entry
$manifest | ConvertTo-Json -Depth 5 | Set-Content $manifestPath -Encoding UTF8

Write-Host "Added $PackageName to Custom tab."
Write-Host "Path: $($entry.path)"

Implement an enhancement for the Windows portable maintenance toolkit: add a “Custom” tab that can add and manage tools using Chocolatey package IDs.

Goal:
Create a feature that lets me add portable or semi-portable tools from Chocolatey into the toolkit, then display them on a new “Custom” tab with launch/update/remove actions.

Important Chocolatey behavior:
- Do not assume Chocolatey can install directly into the toolkit folder.
- Chocolatey packages are tracked under Chocolatey’s package location, commonly $env:ChocolateyInstall\lib\<packageId>.
- Use Chocolatey to resolve/download/install packages, then copy or register the resulting runnable files into the toolkit’s Custom area.
- Prefer known portable packages first.
- Warn clearly when a package appears to be a system installer rather than a portable package.

Add this folder structure if missing:

ToolkitRoot\
  Custom\
    <PackageId>\
  manifests\
    custom-tools.json
  cache\
    chocolatey\

Add a new GUI tab:
Name: Custom

Custom tab features:
1. List custom tools from manifests\custom-tools.json
2. Columns:
   - Name
   - Package ID
   - Version
   - Source
   - Launch Path
   - Installed Date
   - Portable Status
3. Buttons:
   - Add from Chocolatey
   - Add from Folder
   - Launch
   - Open Folder
   - Update
   - Remove
   - Refresh

Add from Chocolatey workflow:
1. Prompt for Chocolatey package ID.
2. Validate choco.exe exists in PATH.
3. Run:
   choco info <packageId> --limit-output
4. Show package info and ask for confirmation.
5. Check package name heuristics:
   - Strong portable signal:
     package ID ends with .portable
     package ID contains portable
     package tags or title contain portable, zip, standalone, commandline, cli
   - Risk signal:
     package mentions install, installer, service, driver, shell extension, context menu, reboot
6. If risky, show warning:
   “This Chocolatey package may install system-wide. Continue?”
7. Install or upgrade with:
   choco install <packageId> --yes --no-progress
8. Locate package files from likely paths:
   - $env:ChocolateyInstall\lib\<packageId>
   - $env:ChocolateyInstall\bin
   - C:\ProgramData\chocolatey\lib\<packageId>
   - C:\ProgramData\chocolatey\bin
9. Detect launchable files:
   - Prefer EXEs under the package’s tools folder
   - Ignore uninstallers, helpers, shims, install scripts, chocolateyInstall.ps1, chocolateyUninstall.ps1
   - If multiple EXEs exist, let the user choose the main executable.
10. Copy the package’s portable/runnable files into:
    ToolkitRoot\Custom\<packageId>\
    unless the package is already self-contained and safe to launch from Chocolatey lib.
11. Write/update manifest entry.
12. Refresh Custom tab.

Manifest format:
Create manifests\custom-tools.json like this:

{
  "tools": [
    {
      "name": "Notepad++ Portable",
      "packageId": "notepadplusplus.portable",
      "source": "Chocolatey",
      "version": "8.x.x",
      "portableStatus": "LikelyPortable",
      "installPath": ".\\Custom\\notepadplusplus.portable",
      "launchPath": ".\\Custom\\notepadplusplus.portable\\notepad++.exe",
      "arguments": "",
      "installedDate": "2026-06-10T10:50:00",
      "lastUpdated": "2026-06-10T10:50:00",
      "notes": ""
    }
  ]
}

Portable status enum:
- LikelyPortable
- PossiblyPortable
- SystemInstallLikely
- Manual

Add from Folder workflow:
1. Let user pick a folder.
2. Scan for EXE files.
3. Let user select main executable.
4. Copy or register folder under Custom.
5. Add manifest entry with source = Manual and portableStatus = Manual.

Launch workflow:
- Resolve relative launchPath from ToolkitRoot.
- Start process with working directory set to executable folder.
- Support optional arguments from manifest.

Update workflow for Chocolatey entries:
1. Run:
   choco outdated --limit-output
2. For selected tool:
   choco upgrade <packageId> --yes --no-progress
3. Re-scan package files.
4. Update copied Custom folder if needed.
5. Update version and lastUpdated in manifest.

Remove workflow:
1. Remove entry from manifest.
2. Ask whether to delete ToolkitRoot\Custom\<packageId>.
3. Do not automatically uninstall system Chocolatey package unless user explicitly chooses:
   “Also uninstall Chocolatey package”
4. If chosen, run:
   choco uninstall <packageId> --yes --no-progress
